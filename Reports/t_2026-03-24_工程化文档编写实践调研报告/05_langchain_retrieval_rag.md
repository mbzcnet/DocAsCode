## LangChain 对“Retrieval”的定位

LangChain 把 Retrieval 视为弥补 LLM 两个天然限制的基础能力：

- 上下文窗口有限：无法一次性塞入整套知识库
- 训练知识静态：模型训练后知识冻结，不能即时覆盖你的最新资料

因此需要在“查询时”检索相关外部信息，把检索结果作为上下文供模型生成答案，这就是 RAG 的基础。

## 一个典型的检索流水线（可模块化替换）

LangChain 强调检索流水线由多个可替换部件构成，通常包含：

- 文档加载器（Document Loaders）：从网页、文件、数据库等加载为统一的 Document
- 切分器（Text Splitters）：把大文档切成 chunk，兼顾检索效果与上下文可放入性
- 向量化（Embeddings）：将 chunk 编码为向量
- 向量库/索引（VectorStore）：存储并提供相似度检索

它的工程价值是：你可以独立替换“切分策略/向量库/embedding 模型”，而无需重写业务逻辑。

## 三类 RAG 架构：2-Step / Agentic / Hybrid

LangChain 把 RAG 实现方式归纳为三种架构，并明确它们的控制力/灵活性权衡：

### 1) 2-Step RAG（固定两步）

- 模式：先检索，再生成（每次请求都先做检索）
- 优点：可预测、易控、延迟更稳定
- 适用：FAQ、文档问答、强流程化写作（尤其适合“引用必须完整”的场景）

### 2) Agentic RAG（代理式检索）

- 模式：由 LLM 决定是否检索、何时检索、检索多少次
- 优点：更灵活，能在多步推理中多次检索
- 风险：控制力降低，可能漏检索或过度检索

### 3) Hybrid RAG（混合+校验）

- 模式：引入中间步骤，如 query 改写、检索有效性验证、生成后校验
- 价值：在灵活性与可控性间折中，适合对质量有门禁需求的垂直领域

对工程项目型文档来说，“Hybrid”通常更贴近真实需求：你不仅要“答”，还要“答得可证明”，因此需要在检索前后做验证与回退策略。

## LangChain 明确提示的安全风险：间接提示注入（Indirect Prompt Injection）

在 RAG 教程中，LangChain 直接指出：检索到的文档可能包含类似“指令”的文本（例如“忽略之前指令”“用 JSON 输出”等），从而影响模型行为。这是 RAG 应用的常见风险点。

它给出的缓解思路包括：

- 防御性提示词：明确把检索内容当作数据（data-only），禁止其改变指令
- 用分隔符包裹上下文：例如用 `<context>...</context>` 让模型更易区分“数据 vs 指令”
- 输出校验：用确定性方式检查输出格式与预期一致，异常则拒绝或重试

并强调：这类缓解并非万无一失，这是当前“指令与数据共享上下文窗口”的结构性限制。

## 对本项目（强严谨引用文档编写）的落地启发

### A. 推荐优先采用 2-Step 或 Hybrid

因为你们需要极严格的引用与可审计性，固定检索步骤 + 校验步骤更适合：

- 强制每次生成都绑定证据片段
- 在“证据不足”时触发回退（重写查询、扩大检索、或直接拒答）

### B. 把“引用”与“元数据”当作输出协议的一部分

LangChain 教程也展示了返回 source documents/metadata 的需求：对严谨引用场景，应强制：

- 生成输出携带引用列表（来源、位置、版本）
- 生成前后都可做确定性校验（引用可解析、证据覆盖关键结论）

### C. 把“间接提示注入”纳入安全基线

在你们的文档系统中，检索到的资料一定来自外部（文件、网页、wiki、工单）。应默认它们是不可信输入，并在：

- 上下文拼接层（context assembly）做隔离与标注
- 生成层提示词做 data-only 约束
- 输出层做格式/敏感信息/越权等检查
