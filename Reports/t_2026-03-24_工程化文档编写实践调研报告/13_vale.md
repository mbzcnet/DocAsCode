## Vale 是什么（官网表述的要点）

Vale 是一个开源的命令行工具，用“像 lint 代码一样 lint 文本”的方式，把写作风格指南落到可自动化执行的规则上。

官网强调了几个对工程文档很关键的特性：

- 离线运行：内容不需要上传到服务器处理（对涉密/内网资料更友好）
- 可定制：支持实现或自定义风格指南规则
- 多平台与多集成：可集成到编辑器与 CI（例如 VS Code、GitHub Actions 等）

## 仓库 README 中强调的关键能力

### 1) Markup-aware（理解标记语言与代码块）

Vale 能理解多种标记格式（Markdown/AsciiDoc/reStructuredText/HTML/XML 等），可以避免对代码块、语法结构产生大量误报。

对工程项目型文档而言，这是“能不能大规模启用 lint”的分水岭：如果误报太多，团队很快会放弃门禁。

### 2) 高可扩展规则系统（用规则执行风格治理）

README 强调 Vale 能执行标准风格指南或企业内部自定义规则（通常以配置/规则文件形式维护）。

这为你的“引用极严谨”目标提供了一个非常实用的落点：除了语法/风格，还可以把“引用规范”写成规则：

- 首次出现缩写必须展开
- 参数/字段必须链接到参考页
- 关键结论必须带引用（citation required）
- 引用格式必须可解析（例如证据对象 id 或固定语法）

### 3) 易安装（独立二进制）

README 强调 Vale 是独立二进制，不需要额外语言运行时；这对 CI 与内网环境部署很友好。

## 对本项目的落地价值（把“写得对”变成门禁）

你们的目标不是“写得好看”，而是“写得对、且不能错”。Vale 的作用在于：

- 把一致性问题（术语、格式、表达）变成自动化门禁
- 降低人工评审负担，让评审聚焦“事实正确性/引用正确性”
- 与 Docs-as-Code 流程自然契合：PR/MR 合并前自动检查

在“AI 参与写作”的场景里，它还能承担一部分“输出后置校验”的职责：当 AI 生成的文本违反规范时直接阻断或要求修订。

## 建议的集成方式（工程实践提示）

- 本地：编辑器保存时或提交前运行，尽早发现问题
- CI：对变更文件运行 Vale，作为合并门禁
- 规则治理：将术语表、风格规则、引用规则作为版本化资产（和文档一起在仓库里演进）
