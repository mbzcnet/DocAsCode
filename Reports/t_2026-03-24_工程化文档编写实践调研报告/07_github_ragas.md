## Ragas 是什么（从 README 可读到的定位）

Ragas 将自己定位为“LLM 应用评测与优化工具包”，强调用客观指标、测试集生成与数据驱动反馈来替代纯人工主观评审。

它的核心诉求是：让评测更快、更可重复，能纳入持续迭代的工程闭环。

## 关键能力点

### 1) 指标化评测（Objective Metrics）

README 强调它提供“预构建指标”，用于常见评测任务；并支持 LLM-based 与传统指标结合。

对工程项目型文档系统而言，建议重点关注可映射到“引用严谨性”的指标维度：

- 上下文相关性：检索出的证据是否相关
- 忠实度/扎根性：回答是否严格基于证据
- 答案相关性：回答是否解决问题本身

### 2) 测试数据生成（Test Data Generation）

Ragas 提到可以生成“生产对齐”的测试集（production-aligned test set generation）。这对你的场景很关键：

- 工程文档常覆盖大量长尾问题（边界条件、异常流程、版本差异）
- 全靠人工维护测试集成本高

因此“自动/半自动”生成覆盖面更广的评测样本，有助于把质量门禁做得更稳。

### 3) 快速脚手架（quickstart templates）

README 提供 `ragas quickstart` 以及 `rag_eval` 模板，用于快速创建评测项目骨架。工程价值在于：

- 能标准化评测工程结构（数据、指标、报告）
- 更容易把评测接入 CI（每次改动跑一遍）

### 4) 可扩展的自定义指标

示例中展示了用一个离散指标（DiscreteMetric）通过自定义 prompt 来做“summary_accuracy”评估。

对“严谨引用”系统，你们很可能需要自定义评测，例如：

- 引用覆盖率是否达标（关键结论是否都有引用）
- 引用可解析性（引用对象是否存在/是否越权/是否定位到具体片段）
- “无证据拒答”是否被正确触发

## 对本项目的落地建议

### A. 把 Ragas 当成“质量门禁引擎”，而不仅是离线分析工具

建议目标是：任何影响检索/切分/索引/提示词/模型版本的变更，都能通过一组固定指标做回归对比，输出可审计报告。

### B. 先定义“你们的评测数据结构”，再选指标

工程项目型文档与通用问答最大的差异是“必须引用”。因此评测样本除了 question/answer/context，还应该包含：

- 期望引用证据（证据对象列表或证据类型约束）
- 版本语境（产品版本、数据截止日期）
- 权限语境（用户角色/租户）

在此基础上再选择/扩展指标，会更贴近你们的真实风险。
