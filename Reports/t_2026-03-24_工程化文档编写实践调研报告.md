# 工程化文档编写实践调研报告（AI 与相关技术）

日期：2026-03-24  
主题：面向工程项目型文档的 AI 工程化编写（多资料、多上下文、强引用严谨性）

## 1. 问题陈述与目标

工程项目型文档往往具备以下特征：

- 资料源多且异构：代码仓库、接口定义、会议纪要、需求/工单、PDF/图纸/表格、数据报表等
- 上下文关联紧密：同一概念跨多处反复出现、版本与环境差异显著
- 引用要求极严：任何引述、数值、结论、接口字段、流程步骤都必须可追溯、可复核、可审计

本调研聚焦“如何通过 AI + 工程化方法”实现：

- 可规模化的资料采集、清洗、结构化、索引与检索
- 以证据为中心（grounded）的生成式写作：带引用、可回溯、无证据则拒答
- 文档质量门禁：格式、链接、术语、版本一致性、可读性、安全合规
- 文档与工程协同（Docs-as-Code / DocOps）：在 Git/CI 的治理下持续迭代

## 2. 现有实践全景（从“写文档”到“文档工程”）

### 2.1 Docs-as-Code / DocOps：把文档当作可测试、可审计的软件产物

“Docs as Code”强调使用与代码一致的工具链与流程：版本控制、评审、自动化测试与发布，从而让文档变更可追溯、可回滚、可在合并前被质量门禁拦截。[^writethedocs_docs_as_code]

典型做法：

- 文档源文件：Markdown / reStructuredText / AsciiDoc 等纯文本
- 变更流程：分支开发 → PR/MR 评审（含领域专家/维护者）→ 合并门禁
- 文档 CI：构建站点/产物 → 链接检查/拼写/术语/风格 → 生成预览环境 → 发布
- 单一事实来源（SSOT）：API/Schema/配置/参数等“参考类内容”尽量从规范文件生成（如 OpenAPI），减少手工复制的漂移风险

### 2.2 信息架构：用 Diátaxis 把内容按用户需求分区

Diátaxis 将文档分为四类：Tutorials（教程）、How-to（操作指南）、Reference（参考）、Explanation（解释/背景），并建议围绕用户不同需求来组织与写作。[^diataxis_home]

对于工程项目型文档，这种分区有两个直接收益：

- 降低“同一篇文档混杂目标”的认知负担
- 便于 AI 生成时采用不同模板与约束（例如 Reference 必须更“干”且可验证）

### 2.3 RAG：以检索增强生成，解决“上下文不足 + 幻觉”

RAG（Retrieval-Augmented Generation）通过在推理时检索外部资料并与提示词一起输入模型，提升对“你自己的数据”的覆盖与可追溯性。该方向的经典工作指出：引入显式的非参数化记忆有助于提供可检查的知识来源，并缓解纯参数记忆的更新与溯源困难。[^rag_paper]

工程实践中，RAG 的关键不在“能答”，而在“可证明地答对、答得可审计”。

### 2.4 评测与可观测：从“感觉不错”变成“可量化改进”

RAG/Agent 系统天然多步骤、多调用、可变路径，需要：

- 评测：度量检索是否相关、回答是否忠实、是否引用充分等（例如 RAGAs 提供无参考的多维指标与流程评估）。[^ragas_acl][^ragas_repo]
- 可观测：以 trace/span 记录每次检索、重排、生成、拒答等关键路径（OpenTelemetry 的 Traces/Spans 是跨系统追踪的标准化语义基础）。[^otel_traces]
- 工具化：例如 TruLens 强调通过指标（groundedness、context relevance 等）系统化评估与迭代，而非“vibe-check”。[^trulens_repo]

### 2.5 安全与治理：把 LLM 视为新的“输入输出边界”

面向工程化文档与知识库应用，安全风险并不只来自模型本身，更来自：

- 间接提示注入（retrieved content 中混入指令）
- 敏感信息泄露（在回答中输出不该出现的内容）
- 过度代理（agent 能执行高危动作）
- 供应链/插件风险等

OWASP LLM Top 10 对上述风险给出了系统性分类与治理建议，可作为落地的风险清单与检查框架。[^owasp_llm_top10]

## 3. 面向“强严谨引用”的 AI 文档工程：目标能力拆解

下面按“严谨引用”核心诉求，拆解系统必须具备的能力：

### 3.1 证据对象化（Evidence as First-Class Citizen）

系统生成任何可陈述事实时，都需要绑定“证据对象”：

- 来源定位：URI/仓库路径、commit/tag、页码/段落/表格坐标
- 内容快照：文档指纹（hash）、切分后 chunk id
- 版本语境：产品版本、环境（dev/stage/prod）、时间范围
- 权限语境：ACL/租户/密级标签（用于检索过滤与展示控制）

目标：任何一句话都能回答“它依据哪一段原始材料？”

### 3.2 生成约束：无证据则拒答（Grounded Generation）

在提示词与输出协议层面强制：

- 只允许基于检索到的证据回答
- 输出必须包含引用列表（每个关键结论至少 1 个证据）
- 证据不足时返回“无法从现有资料证明”，并建议补充资料

这在工程上通常比“让模型更聪明”更有效：把正确性问题转化为“证据可用性/检索质量”问题。

### 3.3 可验证门禁：把“严谨性”变成可自动化测试的规则

将文档质量拆成可自动化检查的门禁项（可在 CI 中强制）：

- 引用完整性：关键段落是否带引用；引用是否能解析到证据对象
- 链接有效性：站内/站外链接可达（或白名单化）
- 术语一致性：术语表/缩写表一致，避免同义多写
- 风格一致性：写作规范与禁用表达（可用 Vale 做本地/CI lint）。[^vale_home]
- 版本一致性：同一文档内出现的版本号/参数是否一致；跨文档是否与 SSOT 对齐

## 4. 可落地的参考架构（“文档工程 + 知识工程 + 生成式编写”）

### 4.1 数据流与组件

可将系统拆成“离线管道 + 在线服务 + 治理与观测”三层：

1) 离线：资料接入与索引（Ingest → Parse → Normalize → Chunk → Embed/Index）  
2) 在线：写作/问答/生成（Retrieve → Rerank → Synthesize → Cite → Validate）  
3) 治理：评测、审计、安全、权限、变更追踪（Eval/Trace/ACL/Policy）

其中 RAG 工程会反复出现的阶段，可参考框架型项目对 RAG 的分解：加载、索引、存储、查询、评估。[^llamaindex_rag]

### 4.2 资料接入（Connectors）与解析（Parsing）

工程项目型资料典型难点是“格式复杂且结构信息重要”（表格、层级标题、编号体系、图纸标注等）。解析层需要尽量保留结构，以便：

- chunk 以语义单元而非按字数硬切
- 引用可定位到页/表/段

解析/ETL 方向的开源方案示例：Unstructured 提供将 PDF/HTML/Office 等转为结构化元素与元数据的能力，可作为接入层或解析层选项之一。[^unstructured_repo][^unstructured_overview]

### 4.3 检索与生成（RAG Orchestration）

RAG 的实现通常包含：

- 索引：向量索引（embeddings）+ 元数据过滤（版本/权限/产品线）+（可选）关键词/BM25 混合检索
- 检索：TopK →（可选）重排（rerank）→ 上下文拼接（含证据边界）
- 生成：带结构化输出协议的合成（例如“结论-依据-引用”三段式）

LangChain 的文档将检索视作 RAG 的基础构件：通过文档加载器、向量库等构建知识库，在运行时检索并把上下文提供给模型。[^langchain_retrieval]

### 4.4 引用生成与校验（Cite & Validate）

建议采用“引用优先”的输出结构：

- 每个小节先列证据清单（证据对象的 id/来源/版本）
- 再生成正文，但正文中的关键陈述必须链接到证据对象
- 生成后跑一次自动校验：引用是否存在、是否越权、证据是否覆盖关键数值与断言

## 5. 关键工程策略（强严谨引用场景的“硬约束”）

### 5.1 版本化与可追溯（SSOT + Git 语义）

对于工程文档，最常见的差错来自“复制粘贴 + 版本漂移”。建议：

- 参考类内容优先从规范生成（API/Schema/配置清单）
- 任何“事实”必须携带版本语境（产品版本、数据截止日期、环境）
- 证据对象绑定到仓库 commit/tag 或文档 hash，避免“源文件变了但引用没变”

### 5.2 权限感知检索（ACL-Aware Retrieval）

知识库/资料库常含内部敏感信息。必须在检索阶段做权限过滤，而不是仅在生成阶段“提醒模型别说”。（模型不应承担访问控制职责）

### 5.3 提示注入防护与输出处理

工程化文档系统的常见攻击面之一是“间接提示注入”（把恶意指令藏在被检索的资料里）。防护要点可参考 OWASP LLM Top 10（Prompt Injection、Insecure Output Handling、Sensitive Information Disclosure 等）。[^owasp_llm_top10]

工程侧可落地措施：

- 明确区分“外部资料”为数据（data-only），禁止其改变系统指令
- 生成输出后进行确定性校验/过滤（例如 JSON schema 校验、敏感信息扫描）
- 高风险操作必须 human-in-the-loop（例如发布到生产、修改规范、生成合规声明）

### 5.4 评测闭环：离线基准 + 线上回归

建议建立“文档正确性基准集”：

- 覆盖典型问题：参数解释、流程步骤、边界条件、数值引用、跨文档一致性
- 每个问题配：可接受答案范围 + 必须引用的证据集合（或证据类型）

再结合评测框架（例如 RAGAs）做持续回归，避免“模型/索引/切分策略改了就退化”。[^ragas_acl]

### 5.5 可观测与审计：Trace 每一次引用从何而来

把一次“文档生成/改写”视为可追踪的流水线：

- 检索阶段：查询、过滤条件、命中文档、重排结果
- 生成阶段：使用了哪些证据对象、哪些陈述引用了哪些证据
- 校验阶段：失败原因（引用缺失/越权/证据不足/格式不合规）

OpenTelemetry 的 traces/spans 能提供跨服务的一致追踪语义，可作为采集标准。[^otel_traces]

## 6. 工具与选型清单（按能力维度）

以下按“能力模块 → 常见选择”罗列，便于后续落地评估：

- 文档工程（Docs-as-Code）：Git + PR/MR + CI 门禁（Docs-as-Code 方法论参考）。[^writethedocs_docs_as_code]
- 信息架构：Diátaxis（四象限内容模型）。[^diataxis_home]
- 解析与结构化：Unstructured（复杂文档转结构化元素）。[^unstructured_repo]
- RAG 编排框架：LlamaIndex（RAG 阶段/概念分解）、LangChain（检索与 RAG 架构拆分）。[^llamaindex_rag][^langchain_retrieval]
- 评测：RAGAs（RAG 管线多维指标）。[^ragas_acl]
- 观测：TruLens（指标化评估与迭代）、OpenTelemetry（标准化 traces）。[^trulens_repo][^otel_traces]
- 文档风格/术语门禁：Vale（离线、可在编辑器与 CI 集成）。[^vale_home]
- 安全治理：OWASP LLM Top 10（风险清单与治理框架）。[^owasp_llm_top10]

## 7. 推荐落地路径（从 0 到可生产）

### 阶段 A：PoC（可证明“引用严谨”）

- 选择 1~2 类高价值文档（例如接口参考 + 运维手册）
- 建立证据对象模型（source/version/page/chunk/acl）
- 实现“生成必须带引用 + 无证据拒答”的最小闭环
- 引入最小门禁：引用解析、链接检查、敏感词/密级扫描

### 阶段 B：MVP（可规模化）

- 增量索引与版本化（与 Git/tag 对齐）
- 混合检索（向量 + 关键词）与重排
- 建立评测集并接入回归评测（每次迭代可量化）
- 引入 trace/审计日志（可定位每个引用的来源与链路）

### 阶段 C：生产（可治理、可合规）

- ACL-aware 检索与多租户隔离
- 安全加固（提示注入、输出处理、过度代理等按 OWASP 清单逐项治理）[^owasp_llm_top10]
- 人工审核流程（对外发布/合规声明/关键数值变更等高风险项）
- 文档门禁体系完善（风格、术语、结构、版本一致性、引用覆盖率）

## 8. 结论

面向工程项目型文档，“AI 能写”并不是难点；难点是“可审计地写对”。当前业界可行的路径是把文档工作拆成工程化流水线（Docs-as-Code/DocOps）与知识工程流水线（RAG/评测/观测/治理），以证据对象与门禁规则把“严谨性”外化为系统约束与自动化检查。
